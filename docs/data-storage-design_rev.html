Data Storage Design

何を・どこに保存するか

Flutter BPR ワークスペース — グループセッション向け設計メモ  |  2026-06-06

主要な判断（5問）
Q1
単独利用 vs チーム共有？
✅ チームで共有しながら使う
BPR は情シス・各部門マネージャ・役員など複数の主語が登場するため、全員がリアルタイムで同じ進捗を参照できることが必要。JSON+Git では Push しないと他の人に見えない。
Q2
頻繁に変わるデータはどれか？
✅ bprTasks・contextNotes・bizItems
BPRタスクは毎週ステージが進み、メモは随時追記され、要チェック業務は担当者が増減する。逆に systems・managementPolicies・workspace は設定値に近く変化が少ない。
Q3
systems（業務システム台帳）はどこに？
✅ DB に入れる（整合性優先）
当初「変化が少ないので JSON でよい」と考えたが、biz_sys_links や policy_system_links が sys_id を参照するため、外部キー制約が張れない問題が発覚。整合性を優先して DB に移動。
Q4
scorecards（ステージ記録）はどう持つか？
✅ 別テーブル（attachments は JSONB）
「特定ステージの記録だけ取得」「日付でソート」などの将来クエリに備えて別テーブルが有利。ただし添付ファイルはまだ実体がないのでメタ情報を JSONB に埋め込む。
Q5
managementPolicies の appliesTo 配列は？
✅ 中間テーブルに分離
JSONB 配列だと「この経営方針に紐づくタスク一覧」を取得するたびアプリ側でパースが必要。中間テーブルなら INSERT/DELETE で単純に管理でき、JOIN クエリも書きやすい。
全体アーキテクチャ図

JSON はリポジトリから直接読み込み、動的データは Neon DB に問い合わせる

保存先の分類
💡 設計方針の総括： ワークスペースのコンセプト設定（名前・アイコン）のみリポジトリで管理し、 1:N・N:M などテーブル間の結合を持つデータ構造はすべて DB 化する。 今回のアプリケーションでは workspace.json を除くほぼすべてのデータがテーブル管理に適切 と判断した。
JSON のまま（リポジトリ）
  • workspace.json ワークスペース名とアイコンの2フィールドのみ。業種変更用の設定値として機能する。
Neon DB（動的データ）
  • departments 本部グループ（bizItems.json の上位層）
  • biz_items 要チェック業務。flags は JSONB
  • bpr_tasks BPRタスク本体。profile をフラット展開・リネーム
  • scorecards ステージ記録。attachments は JSONB で埋め込み
  • systems 業務システム台帳（外部キー整合性のため JSON から移動）
  • biz_sys_links 要チェック業務 ↔ システムの n:m
  • context_notes 空中視点メモ（法令リスク・内部統制・慣行・NG事例）
  • management_policies 経営方針本体
  • policy_task_links 経営方針 ↔ タスク（中間テーブル）
  • policy_system_links 経営方針 ↔ システム（中間テーブル）
テーブル関連図（DB）

実線 = 外部キー（FK）　破線ボックス = 中間テーブル　紫破線 = ポリモーフィック参照（refType で切り替え）
